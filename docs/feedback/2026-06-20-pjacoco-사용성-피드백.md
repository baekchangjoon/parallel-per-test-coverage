# pjacoco 사용성 피드백 (소비자 관점)

- 작성일: 2026-06-20
- 작성자 맥락: `~/github_*` 하위의 **빌드 가능한 Java(Maven/Gradle) 프로젝트 전반**에 pjacoco
  **v1.1.0**(공개 릴리스 `jacocoagent-parallel-1.1.0.jar`)를 부착해 테스트 커버리지를 측정하는 작업을
  수행하면서 실제로 부딪힌 마찰점을 정리한다.
- 측정 방식: 빌드 파일을 **수정하지 않고** 비침습적으로 부착 — Maven은 `-DargLine`, Gradle은
  init script(`tasks.withType(Test) { jvmArgs "-javaagent:…=…,aggregate=true" }`)로 주입하고,
  산출된 `aggregate.exec`를 `jacococli 0.8.13`으로 리포트.

이 문서는 **톤·칭찬이 아니라 재현 가능한 사실**을 우선한다. 좋은 점도 근거와 함께 짧게 적고, 개선점은
증상 → 재현 → 근거 → 제안 순서로, 영향도(P1~P3) 라벨을 붙였다.

---

## 0. 한 줄 요약

aggregate 산출물이 **vanilla JaCoCo와 byte 단위로 동일**하다는 점(검증됨)은 강력한 가치다. 다만
**"평범한 단위테스트 커버리지를 측정한다"는 가장 흔한 진입 시나리오에서 즉시 막히는 P1 결함 두 개**가
있다: (1) **문서상 기본값 `includes=*`가 JVM을 크래시**시키고, (2) **권장 진입 경로(플러그인/테스트킷)가
공개 저장소에 publish되어 있지 않다.** 이 둘만 해소되면 "vanilla 대체 + 필요할 때 per-test"라는
세일즈 포인트가 그대로 산다.

---

## 1. 잘 동작한 점 (근거 포함)

- **aggregate ≡ vanilla (정확 일치).** spring-petclinic에서 pjacoco aggregate와 vanilla
  jacoco 0.8.13을 **동일 조건(같은 includes·classfiles)** 으로 비교했더니 숫자가 자릿수까지 동일했다:
  Instruction `2261/2899`, Line `545/660`, Method `186/220` (78.0% / 82.6% / 84.5%) — 양쪽 완전 일치.
  "기존 JaCoCo·Sonar 생태계 재사용" 주장이 실측으로 확인된다.
- **비침습 부착이 잘 된다.** Maven surefire(`argLine`)·Gradle `Test` 태스크 모두 빌드 파일 수정 없이
  `-javaagent` 한 줄로 붙었고, JUnit4/5 zero-touch가 별도 설정 없이 동작했다.
- **호스트 프로젝트의 jacoco 버전과 무관.** petclinic은 jacoco `0.8.14`(README가 광고하는
  0.8.11–0.8.13 범위 밖)를 쓰는데도 aggregate 모드는 문제없이 측정됐다 — 에이전트가 자체
  jacoco-core를 임베드(shaded)하기 때문. (이 사실 자체를 문서에 명시하면 안심 포인트가 된다 — §2.5)

## 2. 개선점

### P1-A. 문서상 기본값 `includes=*` 가 premain에서 JVM을 크래시시킨다 ★최우선

- **증상.** `includes=*`(옵션 표에 적힌 **기본값**) 또는 `includes`를 **생략**하면, 대상 JVM이 기동
  단계(premain)에서 네이티브 어서션으로 죽는다. 일반 빌드에서는 surefire가
  `"The forked VM terminated without properly saying goodbye … Process Exit Code: 134"`로만 보여줘서
  원인 파악이 매우 어렵다.
- **재현(빌드도 필요 없음, `java -version`만으로 결정적 재현):**

  ```bash
  # (A) 명시적 includes=*  → 크래시
  java -javaagent:jacocoagent-parallel-1.1.0.jar=destfile=/tmp/x,includes=*,aggregate=true -version
  # (C) includes 생략(= 문서상 기본값 *) → 동일 크래시
  java -javaagent:jacocoagent-parallel-1.1.0.jar=destfile=/tmp/x,aggregate=true -version
  ```

  출력:

  ```
  Exception in thread "main" *** java.lang.instrument ASSERTION FAILED ***:
    "!errorOutstanding" … invokeJavaAgentMainMethod at JPLISAgent.c line: 619
  *** java.lang.instrument ASSERTION FAILED ***: "result" … agent load/premain call failed at JPLISAgent.c line: 422
  ```

  반면 **스코프된 include는 정상**(exit 0, `[pjacoco] agent installed`):

  ```bash
  java -javaagent:jacocoagent-parallel-1.1.0.jar=destfile=/tmp/x,includes=com.example.*,aggregate=true -version  # OK
  ```

- **근거/해석.** `includes=*`면 에이전트가 **자기 자신·임베드한 jacoco-core/byte-buddy·부트스트랩
  클래스까지 계측 대상에 넣으려다** premain에서 재진입/어서션으로 죽는 것으로 보인다. 프로젝트 자체
  문서도 이 기본값을 **한 번도 쓰지 않는다** — README의 모든 예제와 petclinic 헬퍼는 항상
  `includes=com.example.*` / `includes=org.springframework.samples.petclinic.*`처럼 명시한다.
  즉 **"문서상 기본값이 실사용에서 곧 크래시"** 라는 모순이 있다.
- **제안.**
  1. **기본값을 안전하게.** `includes` 미지정 시 `*`로 두지 말고, 에이전트가 **자기 패키지·임베드
     런타임(`io.pjacoco.*`, 재배치된 jacoco/byte-buddy, `java.*`/`jdk.*`/`sun.*`)을 항상 자동
     제외**한 뒤 나머지를 계측하도록 한다(=`*`도 안전해짐). 어느 include를 줘도 이 자기 제외는
     무조건 선행해야 한다.
  2. **방어적 실패.** 그래도 위험한 패턴이면 premain에서 네이티브 어서션 대신
     `[pjacoco][ERROR] includes=* is not supported; scope to your application packages, e.g. includes=com.foo.*`
     같은 **명확한 메시지로 즉시 중단**한다. 사용자가 `Exit Code: 134`를 디버깅하게 두지 않는다.
  3. **문서.** 옵션 표의 `includes` 기본값을 실제 안전값으로 바꾸고, "단위테스트 측정 시 앱 패키지로
     스코프하라"를 굵게 명시.

### P1-B. 권장 진입 경로(플러그인 + 테스트킷)가 공개 저장소에 없다 ★최우선

- **증상.** README "빠른 시작(권장)"은
  `plugins { id("io.pjacoco.gradle") version "1.1.0" }` + `testImplementation("io.pjacoco:pjacoco-testkit-junit5:1.1.0")`
  를 제시하지만, 이 좌표들은 **Maven Central·Gradle Plugin Portal·GitHub Packages 어디에서도
  resolve되지 않는다.** 내 머신에서는 `~/.m2`(mavenLocal)에만 1.1.0이 있어 로컬에서만 동작했다.
- **근거.** v1.1.0 GitHub 릴리스 자산은 `jacocoagent-parallel-1.1.0.jar`(+`.sha256`)뿐이고,
  `release.yml`은 **agent jar만** 릴리스에 첨부한다. `docs/PUBLISHING.md`를 보면 플러그인/테스트킷의
  Central·Portal publish는 **시크릿 게이트**(`MAVEN_CENTRAL_USERNAME`, `GRADLE_PUBLISH_KEY` 등)이며,
  그 시크릿이 없으면 단계가 통째로 스킵된다 — 즉 인프라(POM 메타·GPG 서명 gating)는 준비됐지만
  **실제 공개 publish는 아직 일어나지 않았다.**
- **영향.** 이 머신 밖의 누구도(=다른 repo의 CI 포함) 권장 경로로 pjacoco를 못 쓴다. "사용성을 위한
  플러그인/테스트킷"이라는 v1.1.0의 핵심 가치가 소비 불가 상태다.
- **제안.**
  1. 최소한 **GitHub Packages(Maven)** 만이라도 release 워크플로에서 자동 publish해 토큰만으로
     resolve되게 한다(자격증명 부담이 Central/Portal보다 훨씬 작다).
  2. publish 전까지는 README 상단에 **"플러그인/테스트킷은 아직 공개 저장소 미배포 — 현재는
     `publishToMavenLocal` 또는 agent jar 직접 부착만 가능"** 을 명시해 기대와 현실의 간극을 없앤다.
  3. agent jar처럼 **테스트킷 jar도 릴리스 자산으로 첨부**하면 Portal 없이도 `flatDir`/직접
     의존으로 쓸 수 있다.

### P2-C. 멀티모듈에서 `aggregate.exec`가 서로 덮어쓴다

- **증상.** zero-touch aggregate 모드에서 모듈마다 **별도 테스트 JVM**이 같은 `destfile` 디렉터리에
  `aggregate.exec`를 쓴다. 멀티모듈 reactor에서는 **마지막 모듈이 앞 모듈을 덮어써** 전체가 아니라
  마지막 모듈 커버리지만 남는다. (per-test `.exec`는 testId로 이름이 갈려 공존하지만, aggregate는
  파일명이 고정이라 충돌한다.)
- **회피책(실사용에서 내가 쓴 것).** 모듈별 destfile 하위 디렉터리를 강제(`destfile=<out>/${project.name}`
  / Maven `${project.artifactId}`)하고, 끝나고 `jacococli merge`로 합쳤다.
- **제안.** aggregate 파일명을 **JVM/모듈 단위로 자동 네임스페이스**(예 `aggregate-<pid>.exec` 또는
  `<module>/aggregate.exec`)하거나, 최소한 멀티모듈 사용 시의 이 sharp edge와 권장 merge 절차를
  문서화한다. Gradle 플러그인이 있다면 모듈별 destfile을 기본으로 잡아주면 가장 깔끔하다.

### P2-D. aggregate-only인데도 control endpoint를 띄우고, 고정 포트가 병렬에서 충돌

- **증상.** per-test 경계가 필요 없는 **순수 aggregate 모드**에서도 에이전트가 control endpoint를
  열려 시도한다. 한 호스트에서 여러 프로젝트의 테스트 JVM을 병렬로 돌리면 **고정 기본 포트가
  충돌**한다:

  ```
  [pjacoco][WARN] failed to start control endpoint: java.net.BindException: Address already in use
  [pjacoco] agent installed (output=/tmp/x, mode=strict)
  ```

  (위는 다른 pjacoco JVM이 같은 포트를 이미 점유한 상태에서 실제로 찍힌 로그다.)
- **제안.**
  1. aggregate-only/in-process 시나리오에서는 **control endpoint를 아예 열지 않도록** 옵션화(예
     `control=false` 또는 `port=0` → 비활성)하고, 그게 기본이 되게 한다. 서버 socket이 필요 없는
     사용자에게 포트 바인딩은 순수 비용·충돌원이다.
  2. 포트를 열어야 한다면 **0(임의 빈 포트) 자동 할당**을 기본으로 해 병렬 충돌을 없앤다.

### P3-E. 실패 표면이 불친절하다 (P1-A의 부산물)

- premain 실패가 `Exit Code: 134` / "VM crash or System.exit called?"로만 노출돼, pjacoco가 원인임을
  알기 어렵다. 에이전트가 **자기 식별 prefix(`[pjacoco][ERROR] …`)로 stderr에 한 줄** 남기고 죽으면
  진단 시간이 크게 준다. (정상 경로의 `[pjacoco] agent installed …` 로그는 좋다 — 실패 경로에도 같은
  수준의 친절함을 적용하자.)

### P3-F. 용어/기본 모드 문서화

- **`destfile`이 실제로는 "파일"이 아니라 "디렉터리"** 다. vanilla jacoco의 `destfile`(단일 파일)과
  의미가 충돌해, jacoco 사용자가 헷갈린다. README에 "출력 디렉터리"라고 적혀는 있으나, 이름 자체가
  오해를 부른다 — 별칭(`destdir=`)을 두거나 주석을 강화하면 좋다.
- 기본 `mode=strict`라 **테스트킷 없이 in-process로 붙이면 per-test 기록은 0**이고 aggregate만 쌓인다.
  "테스트킷 없이 = aggregate 전용(=vanilla 대체)"라는 점을 빠른 시작 바로 옆에 한 줄로 명시하면,
  per-test를 기대했다가 비는 사용자의 혼란을 막는다.

## 3. 제품 제안: "vanilla JaCoCo 드롭인" 1분 퀵스타트를 전면에

per-test·baggage·control-url 중심 서사에 가려져 있지만, 실측상 pjacoco의 **가장 진입장벽 낮은 가치**는
"**vanilla와 동일한 전체 커버리지를, 빌드 수정 없이, 필요하면 per-test까지**"다. 다음 수준의 3줄
레시피를 README 최상단에 두기를 제안한다(아래는 내가 실제로 검증한 형태):

```bash
# 1) 에이전트 부착 (앱 패키지로 스코프)
mvn test -DargLine="-javaagent:jacocoagent-parallel.jar=destfile=cov,includes=com.foo.*,aggregate=true"
# 2) 리포트 (산출 aggregate.exec는 vanilla 포맷)
java -jar jacococli.jar report cov/aggregate.exec --classfiles target/classes --html site
```

여기에 P1-A(기본값 안전화)·P1-B(공개 배포)만 더해지면, "설치 1줄 → vanilla 동일 결과 → 원하면
per-test"로 이어지는 매끄러운 깔때기가 완성된다.

## 4. 우선순위 정리

| ID | 항목 | 영향도 | 한 줄 |
|---|---|---|---|
| P1-A | `includes=*`(기본값) premain 크래시 | **P1** | 자기/런타임 자동 제외 + 방어적 실패 메시지 |
| P1-B | 플러그인/테스트킷 공개 미배포 | **P1** | 최소 GitHub Packages publish, 그 전까진 README 명시 |
| P2-C | 멀티모듈 aggregate.exec 덮어쓰기 | P2 | 파일명 모듈/JVM 네임스페이스 또는 문서화 |
| P2-D | aggregate-only도 control 포트 바인딩·충돌 | P2 | 엔드포인트 옵트아웃 기본화 / 임의포트 |
| P3-E | premain 실패 표면 불친절 | P3 | `[pjacoco][ERROR]` 한 줄 후 종료 |
| P3-F | `destfile`(=디렉터리) 용어·strict 기본 모드 문서화 | P3 | 별칭/주석, 빠른 시작 옆 한 줄 |

> 검증 환경: macOS(arm64), Corretto 17.0.18, pjacoco agent v1.1.0(공개 릴리스), jacococli 0.8.13,
> 대상 = `~/github_*` 빌드 가능 Java 프로젝트들. P1-A/P2-D 로그는 본 작업 중 실제 출력에서 인용.
