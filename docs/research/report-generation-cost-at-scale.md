# 대규모 코드베이스에서의 리포트 생성 비용 — 왜 probe는 싸도 report는 비쌀 수 있는가

- 작성일: 2026-06-13
- 질문: "에이전트와 함께 실행되는 앱의 코드가 **백만 라인** 수준이면, probe 수집은 빨리
  끝나도 생성된 `.exec`로 **리포트**를 만들 때는 터치되지 않은 라인·코드도 모두 표시해야
  하니 오래 걸리지 않나?"
- 결론: **맞다 — 단, 느려지는 지점은 "probe 수집"이 아니라 "분석(Analysis) 단계"이고,
  비용은 커버된 양이 아니라 분석 범위(scope) 전체 크기에 비례한다.** per-test 목적에서는
  분모(전체 라인)를 전역 1회만 계산하고 테스트별로는 **exec에 등장한 클래스만** 분석하면
  코드베이스 크기와 거의 무관하게 빠르게 유지할 수 있다.

## JaCoCo의 두 단계 — 비용이 어디서 발생하나

### 1. 런타임 probe 수집 (싸다)

- 계측된 클래스마다 `boolean[]` probe 배열(`$jacocoData`)이 있고, 실행 시 해당 probe를
  `true`로 세팅하는 게 전부다. 동기화 없는 단순 쓰기 — 의도적으로 최소 오버헤드.
- `.exec` 파일에는 **클래스 ID + 이름 + boolean[]**(어떤 probe가 hit됐는지)만 들어간다.
- 따라서 exec 크기와 수집 비용은 "라인을 터치했는지"가 아니라 **로드된 클래스 수**에
  비례한다. 코드베이스가 커도 **한 테스트가 실제로 건드리는 클래스는 보통 극소수**라
  exec는 작고 수집은 빠르다. (이 프로젝트의 per-test 스토어 복제도 동일하게 additive 쓰기라
  싸다.)

### 2. 리포트 생성 = 분석(Analysis) 단계 (비쌀 수 있다)

- 리포트를 만들 때 JaCoCo `Analyzer`는 **원본 `.class` 바이트코드를 다시 ASM으로 파싱**해
  CFG와 probe→line 매핑을 재구성하고, exec의 실행 데이터를 그 위에 덮는다.
- 결정적으로, **0%인(=터치 안 된) 라인·클래스도 리포트에 "미커버"로 나와야 하므로 분석
  범위에 포함된 모든 클래스를 빠짐없이 파싱**해야 한다.
- 즉 리포트 생성 비용은 **커버된 양이 아니라 분석 범위 전체 크기(=1M 라인)에 비례**한다.
  여기에 HTML 출력이면 소스 파일당 페이지 렌더링까지 더해져 가장 비싸다.

질문의 직관 — "터치되지 않은 라인과 코드도 모두 표시해야 하므로 오래 걸린다" — 은 바로 이
분석 단계를 정확히 가리킨다.

## per-test 시나리오에서의 진짜 함정

per-test로 **테스트마다 풀 리포트를 생성**하면 비용이 이렇게 곱해진다:

```
총비용 ≈ N_tests × (전체 1M 라인 바이트코드 재파싱 + 렌더링)
```

probe/exec는 싸도, **전체 코드베이스 분석을 테스트 수만큼 반복**하는 게 병목이다. 백만 라인
× 수천 테스트면 분석 단계가 지배적으로 커진다.

## 대응책 (효과 큰 순서)

1. **per-test 리포트는 "터치된 클래스만" 분석한다.**
   exec 데이터 스토어에는 그 테스트에서 **실행된 클래스만** 들어 있다. 분모(전체 라인 수,
   미커버 포함)는 **글로벌 상수**이므로 한 번만 계산하고, 테스트별로는 exec에 등장한
   클래스만 `Analyzer`에 넘기면 1M 전수 스캔이 사라진다. → 가장 큰 절감.

2. **구조 분석을 1회만 하고 exec만 갈아끼운다.**
   클래스 구조(probe 위치, line 매핑)는 모든 테스트에서 **동일**하고 다른 건 boolean[]뿐이다.
   표준 `Analyzer`는 매번 재파싱하지만, 바이트코드를 한 번 파싱해 구조를 캐시해 두고
   테스트별 boolean[]만 라인에 오버레이하면 N× ASM 파싱을 제거할 수 있다.

3. **HTML 대신 XML/CSV.** per-test는 보통 "이 테스트가 어떤 라인을 덮었나"라는 구조 데이터만
   필요하므로, 소스당 HTML 페이지 렌더링을 빼면 크게 빨라진다.

4. **분석 scope를 애플리케이션 패키지로 제한**(의존성 jar 제외).

5. **수집과 리포팅을 분리.** 핫패스(테스트 실행 중)에는 per-test `.exec`만 떨어뜨리고(싸다),
   리포트는 오프라인에서 lazy하게 / 필요한 것만 생성한다.

## 정리

- **"전체 리포트(분모에 미커버 라인 전부 포함)를 테스트마다 풀로 생성"** → 1M 라인이면
  매번 전체 분석이라 느리다. 질문의 직관이 맞다.
- 하지만 per-test 목적상 보통 필요한 건 **"이 테스트가 덮은 것"(분자)**뿐이고 분모는 전역
  상수이므로, **exec에 등장한 클래스만 분석**하도록 설계하면 코드베이스 크기와 거의 무관하게
  빠르게 유지된다.
- 핵심 한 줄: **probe 수집 비용 ∝ 실행된 클래스 수, 리포트 분석 비용 ∝ 분석 범위 전체 크기.**
  대규모에서 비용을 통제하려면 per-test의 분석 범위를 "터치된 클래스"로 좁혀라.
